# 第12章：飞书Bot配置

推荐指数：五星。飞书是国内最推荐的IM平台，功能完整、开发友好。

状态：生产就绪，支持机器人私聊和群组，使用 WebSocket 长连接模式接收消息。

## 12.1 飞书机器人介绍

### 12.1.1 为什么选择飞书？

飞书（Lark/Feishu）是字节跳动推出的企业协作平台，凭借其完善的开放能力和友好的开发体验，成为 OpenClaw 最推荐对接的 IM 渠道。

**技术优势**：

- **流式输出**：支持实时流式响应，消息逐字呈现，交互体验最好
- **富文本支持**：支持 Markdown 语法和卡片消息，便于格式化输出
- **API 完善**：开发文档清晰，接口设计规范，版本迭代稳定
- **WebSocket 长连接**：消息实时推送，无需配置公网地址即可接收事件

**功能优势**：

- **多 Agent 支持**：可在同一飞书账号下配置多个专业助手，按需切换
- **群组功能**：支持群聊场景及 @ 提及触发，适合团队共享使用
- **文件传输**：支持文件上传下载，可用于文档处理类任务
- **访问控制**：内置白名单与权限管理机制，支持精细化用户管理

**使用优势**：

- **免费使用**：基础功能完全免费，适合个人开发者快速上手
- **跨平台**：支持 Windows、Mac、iOS、Android 及 Web 端
- **现代化设计**：界面简洁，操作流程符合直觉
- **企业级稳定性**：服务可靠，适合个人和团队长期使用

### 12.1.2 适用场景

飞书 Bot 在以下场景中具有明显优势，可根据实际需求选择合适的部署模式。

个人效率方面，用户可随时随地通过飞书客户端向机器人发起对话，实现随身携带的 AI 助手。团队协作方面，多名成员可共享同一机器人，在群组中协同使用 AI 能力，显著降低信息传递成本。项目管理方面，可借助机器人完成任务跟踪与进度汇报，结合飞书自身的日历和文档功能形成完整工作流。知识管理方面，机器人可辅助文档整理与信息检索，提升团队知识沉淀效率。工作自动化方面，配合定时任务与事件触发，可构建轻量级业务流程自动化方案。

## 12.2 快速开始

### 12.2.1 前置要求

在开始配置之前，需要确认以下三类准备工作已就绪，任何一项缺失都可能导致后续步骤无法正常进行。

**飞书账号**：需要一个已注册的飞书账号（个人账号或企业账号均可）并安装飞书客户端，部分企业账号可能需要管理员授权才能创建自建应用。

**OpenClaw 环境**：需要完成 OpenClaw 的安装（参见第2章安装），并确认 Gateway 服务处于正常运行状态，这是接收飞书消息的基础服务。

**开发者权限**：需要具备在飞书开放平台创建应用和配置机器人的权限。个人账号通常默认拥有该权限；企业账号如无权限，需联系企业管理员开通。

### 12.2.2 配置流程概览

如图12-1所示，完整配置流程共分为五个阶段，依次递进，前一阶段完成后方可进入下一阶段。

```
1. 创建飞书应用
   |
2. 获取 App ID 和 App Secret
   |
3. 配置 OpenClaw
   |
4. 启动并测试
   |
5. 配置访问控制（可选）
```

图12-1  飞书 Bot 配置流程图

整个流程的预计完成时间为 15 至 30 分钟。其中飞书应用的审核发布通常在几分钟内完成，若遇到企业账号需要管理员审核的情况，时间会相应延长。

## 12.3 创建飞书应用

本节介绍在飞书开放平台完成应用创建的完整步骤。这些操作均在飞书开放平台的开发者后台中完成，该平台是飞书为开发者提供的统一管理入口。

### 12.3.1 步骤1：访问飞书开放平台

访问飞书开放平台（搜索"飞书开放平台"即可），使用飞书账号登录后，进入"开发者后台"。开发者后台是管理所有自建应用的核心页面，首次访问时系统会引导完成开发者协议确认。

确认已成功进入开发者后台后，即可开始创建第一个自建应用。

### 12.3.2 步骤2：创建企业自建应用

企业自建应用是将 OpenClaw 接入飞书的标准方式，相比第三方应用拥有更高的权限灵活度和更稳定的 API 访问能力。

在开发者后台点击"创建企业自建应用"，按提示填写以下信息：

- **应用名称**：填写 OpenClaw Bot（或自定义，建议与用途相关，便于识别）
- **应用描述**：填写 AI 助手机器人（简短描述即可）
- **应用图标**：上传图标（可选，不影响功能）

填写完成后点击"创建"，系统将自动生成应用的基础信息并跳转至应用详情页。应用创建成功意味着飞书已为该机器人分配了唯一的标识符，后续所有配置均在此应用下进行。

### 12.3.3 步骤3：获取凭证信息

应用创建完成后，需要获取用于 OpenClaw 认证的凭证信息。在应用详情页中点击左侧菜单的"凭证与基础信息"，找到以下两项关键数据并记录：

- **App ID**：格式为 `cli_xxxxxxxxxx`，是应用的唯一标识
- **App Secret**：点击"查看"后复制，用于接口请求签名

App Secret 仅在查看时短暂显示，建议立即复制并保存到安全的位置。一旦离开该页面，下次查看需要重新验证身份。若不慎丢失，可在同一页面重置 App Secret，但重置后旧的 Secret 立即失效，已配置的 OpenClaw 需同步更新。

### 12.3.4 步骤4：配置应用权限

权限配置决定了机器人能够执行的操作范围。权限过少会导致机器人无法收发消息，权限过多则存在安全风险。下表列出了飞书 Bot 正常运行所需的最小权限集合。

在应用详情页点击左侧菜单的"权限管理"，依次搜索并开通以下权限，见表12-1。

表12-1  飞书应用所需权限列表

| 权限标识 | 说明 | 是否必需 |
|---|---|---|
| `im:message` | 获取与发送单聊、群组消息 | 必需 |
| `im:message.group_at_msg` | 获取群组中所有消息 | 必需 |
| `im:message.group_at_msg:readonly` | 接收群聊中 @ 机器人消息事件 | 必需 |
| `im:message.p2p_msg` | 获取用户发给机器人的单聊消息 | 必需 |
| `im:message.p2p_msg:readonly` | 接收用户发给机器人的单聊消息事件 | 必需 |
| `im:message:send_as_bot` | 以应用的身份发消息 | 必需 |
| `im:chat` | 获取群组信息 | 推荐 |
| `im:chat:readonly` | 获取群组信息（只读） | 推荐 |
| `contact:user.id:readonly` | 获取用户 user ID | 推荐 |

表中必需权限是机器人收发消息的基础，缺少任何一项都会导致消息处理流程中断。推荐权限用于群组管理和用户身份识别，在配置白名单和多群组场景中不可缺少，建议一并开通。

开通权限后，需要发布应用版本才能生效。点击页面右上角的"发布版本"，选择"全员可用"，提交后通常几分钟内完成审核。企业账号如需管理员审核，请联系管理员处理。

### 12.3.5 步骤5：配置事件订阅

事件订阅决定了飞书服务器在何种情况下向 OpenClaw 推送消息通知。若不配置事件订阅，机器人将无法感知用户发送的消息，因此这一步骤不可跳过。

在应用详情页点击左侧菜单的"事件订阅"，再点击"添加事件"，搜索并添加以下事件：

**必需事件**：

- `im.message.receive_v1`：接收消息事件，用于感知用户向机器人发送的单聊消息及群组中 @ 机器人的消息

配置请求地址一栏可暂时留空，OpenClaw 采用 WebSocket 长连接方式主动拉取消息，无需飞书服务器主动回调，因此不需要配置公网地址。这也是飞书 Bot 配置相比其他平台更简便的原因之一。

## 12.4 配置 OpenClaw

飞书应用创建完毕后，需要将 App ID 和 App Secret 填写到 OpenClaw 的配置中，使 Gateway 服务能够以该应用身份连接飞书并接收消息。OpenClaw 提供三种配置方式，按照使用便捷程度依次介绍。

### 12.4.1 方式一：使用配置向导（推荐）

配置向导以交互式问答的形式引导完成配置，适合首次配置或不熟悉配置文件结构的用户。在终端中执行以下命令，按照提示依次完成操作。

```bash
# 运行配置向导
openclaw onboard
```

向导启动后，依次执行以下操作：选择"Configure Channels"，选择"Feishu"，输入 App ID，输入 App Secret，最后确认完成配置。向导会自动将输入的信息写入配置文件并验证格式，完成后可直接进入启动测试阶段。

### 12.4.2 方式二：手动编辑配置文件

手动编辑配置文件适合需要同时调整多项参数的场景，也便于版本控制和配置复用。编辑配置文件 `~/.openclaw/openclaw.json`，在 `channels` 节点下添加飞书配置，代码如下：

```json
// ~/.openclaw/openclaw.json（第12章/openclaw.json）
{
  "channels": {
    "feishu": {
      "enabled": true,
      "appId": "cli_xxxxxxxxxx",
      "appSecret": "your_app_secret_here",
      "verificationToken": "auto_generated",
      "encryptKey": "auto_generated"
    }
  }
}
```

其中 `verificationToken` 和 `encryptKey` 填写 `auto_generated` 表示由 OpenClaw 自动生成，无需手动填写。`appId` 和 `appSecret` 替换为第12.3.3节中获取的实际值。

### 12.4.3 方式三：使用命令行

命令行方式适合在脚本或自动化部署场景中使用，每个配置项单独设置，便于精确控制。在终端中依次执行以下命令：

```bash
# 配置 App ID
openclaw config set channels.feishu.appId cli_xxxxxxxxxx

# 配置 App Secret
openclaw config set channels.feishu.appSecret your_app_secret_here

# 启用飞书频道
openclaw config set channels.feishu.enabled true
```

三条命令执行完毕后，配置即时写入配置文件，无需额外保存操作。可通过 `openclaw config get channels.feishu` 验证配置已正确写入。

## 12.5 启动并测试

配置完成后，需要重启 Gateway 服务使新配置生效，然后通过飞书客户端发送消息验证机器人是否正常响应。

### 12.5.1 重启 Gateway

执行以下命令重启 Gateway 并确认服务状态。重启后需等待几秒让服务完成初始化，再检查运行状态。

```bash
# 重启 Gateway 以应用配置
openclaw gateway restart

# 等待初始化完成后检查状态
sleep 5
openclaw gateway status
```

服务正常运行时，输出应包含以下信息：

```
Runtime: running (pid xxxxx, state active)
RPC probe: ok
Listening: *:18789
Dashboard: http://127.0.0.1:18789/
```

`RPC probe: ok` 表明 Gateway 核心服务已就绪，`Listening` 行确认端口监听正常。如果输出中出现 `error` 或 `failed` 字样，应先查看日志排查原因，再继续后续步骤。

### 12.5.2 在飞书中找到机器人

Gateway 启动成功后，打开飞书客户端，点击底部导航栏的"工作台"，在应用列表中找到已创建的应用（OpenClaw Bot），点击进入即可开始与机器人对话。

若在工作台中未找到应用，说明应用版本尚未完成审核发布，或当前账号不在应用的可用范围内，需要返回飞书开放平台确认发布状态。

### 12.5.3 发送测试消息

在对话框中发送以下内容以验证机器人连通性：

```
你好，能听到我说话吗？
```

### 12.5.4 验证回复

如果配置成功，机器人会回复类似如下内容：

```
你好！我是 OpenClaw AI 助手，很高兴为您服务。
我可以帮您：
- 回答问题
- 处理任务
- 搜索信息
- 执行命令

有什么可以提供帮助的吗？
```

收到回复即表明飞书 Bot 已配置成功，消息收发链路完全打通。如果机器人没有回复，请按照12.12节的常见问题排查步骤进行处理。

### 12.5.5 查看日志

当测试遇到问题时，日志是定位原因的最直接途径。执行以下命令实时查看相关日志输出：

```bash
# 查看 Gateway 日志
tail -f ~/.openclaw/logs/gateway.log

# 查看飞书频道日志
tail -f ~/.openclaw/logs/feishu.log
```

Gateway 日志记录所有消息的收发过程和错误信息；飞书频道日志专门记录与飞书 API 交互的详细请求和响应，两者结合可快速定位问题所在层级。

## 12.6 访问控制

### 12.6.1 为什么需要访问控制？

默认情况下，任何能够找到机器人的飞书用户都可以与其对话，这在公开的企业环境中存在一定风险。访问控制机制通过限定可用用户的范围，从三个维度保护机器人服务。

**安全性**：防止未授权用户访问，避免敏感信息泄露。**成本控制**：每次对话都会消耗后端 AI 模型的 API 调用额度，白名单机制可防止 API 被滥用导致意外费用。**用户管理**：在多人团队中，管理员可精确控制哪些成员有权使用机器人及其权限级别。

### 12.6.2 配置白名单

白名单配置分为用户维度和群组维度两类，可单独使用，也可组合使用。

**用户白名单**仅允许列表中的飞书用户 ID 与机器人对话。用户 ID 的获取方式参见第12.8节。执行以下命令添加允许的用户：

```bash
# 添加允许的用户 ID
openclaw config set channels.feishu.allowlist '["ou_xxxxxx","ou_yyyyyy"]'
```

**群组白名单**仅允许列表中的群组使用机器人，在群组内发送的消息才会被处理。执行以下命令添加允许的群组：

```bash
# 添加允许的群组 ID
openclaw config set channels.feishu.groupAllowlist '["oc_xxxxxx","oc_yyyyyy"]'
```

若需要同时配置用户和群组白名单，也可直接编辑配置文件 `~/.openclaw/openclaw.json`，代码如下：

```json
// ~/.openclaw/openclaw.json（第12章/openclaw.json）
{
  "channels": {
    "feishu": {
      "enabled": true,
      "appId": "cli_xxxxxxxxxx",
      "appSecret": "your_app_secret_here",
      "allowlist": [
        "ou_user1",
        "ou_user2"
      ],
      "groupAllowlist": [
        "oc_group1",
        "oc_group2"
      ]
    }
  }
}
```

配置白名单后，白名单之外的用户发送消息时机器人不会响应，也不会产生任何 API 调用消耗。

### 12.6.3 配置管理员

管理员账号在普通用户权限之上拥有额外的操作能力，包括执行管理命令、查看系统运行状态以及管理其他用户的访问权限。建议将机器人所有者的账号配置为管理员。执行以下命令添加管理员：

```bash
# 添加管理员用户
openclaw config set channels.feishu.admins '["ou_admin1","ou_admin2"]'
```

管理员账号同样需要在用户白名单中，或不配置白名单（即允许全部用户）时才能正常使用管理功能。

## 12.7 群组配置

### 12.7.1 将机器人添加到群组

飞书机器人支持在群组中使用，适合团队共享 AI 能力的场景。将机器人添加到群组共分三步。

第一步，在飞书客户端中创建群组（如果目标群组尚不存在），并将需要使用机器人的成员添加到群组中。群组人数和类型不影响机器人的接入方式。

第二步，打开目标群组，点击右上角的"..."更多菜单，选择"添加机器人"，在搜索框中输入已创建应用的名称（如 OpenClaw Bot），选中后点击"添加"完成操作。添加成功后群组内会出现机器人加入的系统提示消息。

第三步，在群组中测试机器人是否正常响应。发送以下消息验证 @ 提及功能：

```
@OpenClaw Bot 你好
```

机器人正常回复即表明群组配置成功。若无响应，请确认第12.3.4节的群组相关权限已全部开通。

### 12.7.2 群组消息模式

OpenClaw 在群组场景下支持两种消息响应模式，见表12-2。两种模式分别适用于不同的使用场景，根据群组用途选择合适的模式可以显著改善使用体验。

表12-2  群组消息模式说明

| 模式 | 触发条件 | 适用场景 |
|---|---|---|
| @ 提及模式（默认） | 仅 @ 机器人时响应 | 多人群组，按需使用 |
| 全部消息模式 | 响应群组中所有消息 | 专用机器人群组 |

**@ 提及模式**是默认模式，适合普通工作群组，机器人不会打扰正常的群组讨论，只在被主动 @ 时才介入。使用方式如下：

```
@OpenClaw Bot 今天天气怎么样？
```

**全部消息模式**适合创建专用的 AI 对话群组，群组中的所有消息都会被机器人处理并回复。由于该模式会对群组中每条消息都发起 AI 调用，会消耗更多 API 额度，建议仅在专用群组中启用。执行以下命令开启该模式：

```bash
# 启用全部消息模式
openclaw config set channels.feishu.respondToAllGroupMessages true
```

## 12.8 获取群组/用户 ID

在配置白名单和管理员时，需要知道飞书用户和群组的 ID。飞书的用户 ID 格式为 `ou_` 开头，群组 ID 格式为 `oc_` 开头。以下介绍两种获取方式。

### 12.8.1 获取用户 ID

**方法1：通过日志查看**

这是最直接的方式，无需额外工具。先启动日志实时监控，再让目标用户向机器人发送一条消息，日志中会自动记录该用户的 ID。

```bash
# 监控 Gateway 日志
tail -f ~/.openclaw/logs/gateway.log
# 用户发送消息后，日志中将出现：
# Received message from user: ou_xxxxxx
```

记录日志中 `ou_` 开头的字符串，即为该用户的飞书用户 ID。

**方法2：使用命令查询**

如果已知目标用户的手机号，可通过以下命令直接查询其用户 ID，无需用户主动发送消息：

```bash
# 通过手机号查询用户信息
openclaw channels feishu user-info --phone "+8613800138000"
```

该命令会返回用户的基础信息，包括 user ID，将其复制至白名单配置中即可。

### 12.8.2 获取群组 ID

**方法1：通过日志查看**

与获取用户 ID 的思路相同，先启动日志监控，再在目标群组中发送任意消息（包括 @ 机器人），日志中即会记录该群组的 ID。

```bash
# 监控 Gateway 日志
tail -f ~/.openclaw/logs/gateway.log
# 在群组中发送消息后，日志中将出现：
# Received message from group: oc_xxxxxx
```

**方法2：通过飞书开放平台查询**

登录飞书开放平台的开发者后台，选择已创建的应用，进入"机器人"配置页，在"群组列表"中可以查看机器人已加入的所有群组及其对应的群组 ID。该方法适合批量查询多个群组 ID 的场景。

## 12.9 高级配置

本节介绍三类进阶配置项，分别针对消息格式、请求超时和代理网络场景，可按需独立启用，不影响其他配置。

### 12.9.1 配置消息格式

消息格式配置决定了机器人向飞书发送的消息类型。默认情况下使用纯文本格式，开启以下配置后，机器人将以 Markdown 富文本和卡片消息形式输出，界面呈现更为美观。代码如下：

```json
// ~/.openclaw/openclaw.json（第12章/openclaw.json）
{
  "channels": {
    "feishu": {
      "messageFormat": "markdown",
      "enableRichText": true,
      "enableCard": true
    }
  }
}
```

`messageFormat` 设为 `markdown` 后，OpenClaw 输出的 Markdown 语法（如标题、列表、代码块）将被转换为飞书原生的富文本格式。`enableCard` 开启后，特定类型的回复（如搜索结果、结构化数据）将以卡片消息展示，支持按钮交互。

### 12.9.2 配置超时时间

当 AI 模型响应较慢或网络条件不稳定时，适当增大超时阈值可以避免消息处理被提前中断。`timeout` 单位为毫秒，`retryAttempts` 控制请求失败后的重试次数。代码如下：

```json
// ~/.openclaw/openclaw.json（第12章/openclaw.json）
{
  "channels": {
    "feishu": {
      "timeout": 30000,
      "retryAttempts": 3
    }
  }
}
```

默认超时时间为 10000 毫秒（10秒），对于使用长上下文或复杂推理模型的场景，建议调整为 30000 至 60000 毫秒。`retryAttempts` 建议设置为 2 至 3 次，过多重试会导致用户等待时间过长。

### 12.9.3 配置代理

在无法直接访问飞书 API 的网络环境中（如某些企业内网），可通过代理服务器转发请求。代码如下：

```json
// ~/.openclaw/openclaw.json（第12章/openclaw.json）
{
  "channels": {
    "feishu": {
      "proxy": {
        "enabled": true,
        "host": "127.0.0.1",
        "port": 7890
      }
    }
  }
}
```

`host` 和 `port` 填写本地代理服务的监听地址和端口。配置代理后，OpenClaw 与飞书 API 之间的全部通信将通过该代理服务器转发。若代理服务器需要身份验证，还需额外添加 `username` 和 `password` 字段。

## 12.10 多账号配置

### 12.10.1 配置说明与示例

OpenClaw 支持同时配置多个飞书机器人，每个机器人对应独立的飞书应用，拥有各自的 App ID、App Secret 和用户白名单。多账号配置通过在 `accounts` 数组中依次列出各账号信息来实现，代码如下：

```json
// ~/.openclaw/openclaw.json（第12章/openclaw-multi-account.json）
{
  "channels": {
    "feishu": {
      "enabled": true,
      "accounts": [
        {
          "id": "bot1",
          "appId": "cli_xxxxxxxxxx",
          "appSecret": "secret1",
          "allowlist": ["ou_user1"]
        },
        {
          "id": "bot2",
          "appId": "cli_yyyyyyyyyy",
          "appSecret": "secret2",
          "allowlist": ["ou_user2"]
        }
      ]
    }
  }
}
```

每个账号通过 `id` 字段区分，该字段将出现在日志中，便于追踪具体是哪个机器人在处理消息。`allowlist` 可为每个账号单独设置，实现用户的精细隔离。

### 12.10.2 使用场景

多账号配置主要适用于以下三类场景。

**部门隔离**：不同部门使用不同的机器人实例，各部门的对话记录和权限相互独立，避免信息混用。**功能分离**：将工作助手与生活助手部署为不同账号，配置不同的系统提示词和工具集，提升专注度。**用户分组**：VIP 用户与普通用户分配到不同机器人，可针对不同群体配置不同的模型和响应策略。

## 12.11 多 Agent 配置

### 12.11.1 什么是多 Agent？

多 Agent 功能（目前处于新功能阶段）允许在同一飞书账号下配置多个具有不同专长的 AI 助手。用户可以通过不同的触发词或命令切换到对应的 Agent，而无需维护多个飞书应用。

每个 Agent 可以独立配置以下要素：专业领域（如编程、写作、翻译、财务等）；底层模型（如 GPT-4、Claude、Kimi、DeepSeek 等）；专属的系统提示词，决定 Agent 的角色定位和回答风格；以及可调用的工具集和 Skills，决定 Agent 能够执行的操作范围。

多 Agent 的详细配置说明请参见第3章进阶部署中的相关内容，以及附录部分的多 Agent 配置指南和实战案例。

### 12.11.2 快速开始

以下命令展示了创建并配置一个编程助手 Agent 的基本流程：

```bash
# 创建新 Agent
openclaw agent create --name "编程助手" --model "gpt-4"

# 配置 Agent 的系统提示词
openclaw agent config --name "编程助手" --system-prompt "该助手是专业的编程助手"
```

Agent 创建完成后，在飞书对话框中通过以下方式调用：

```
@编程助手 帮我写一个 Python 函数，实现冒泡排序
```

## 12.12 常见问题

### 12.12.1 机器人无法收到消息

**原因分析**：消息接收依赖 Gateway 服务正常运行、飞书应用权限开通以及事件订阅配置三个环节，任何一个环节出问题都会导致消息无法送达。

首先检查 Gateway 状态。如果 Gateway 未运行，则 OpenClaw 无法建立与飞书的 WebSocket 连接，飞书侧的消息将无处转发。

```bash
# 检查 Gateway 运行状态
openclaw gateway status

# 如果未运行，查看错误日志
tail -f ~/.openclaw/logs/gateway.log

# 重启 Gateway
openclaw gateway restart
```

如果 Gateway 状态正常但仍收不到消息，进入飞书开放平台检查第12.3.4节列出的必需权限是否已全部开通，以及第12.3.5节的事件订阅是否已添加 `im.message.receive_v1` 事件。权限变更需要重新发布应用版本才能生效。

### 12.12.2 机器人回复很慢

**原因分析**：响应速度受三个因素影响：AI 模型本身的推理速度、网络请求的往返延迟，以及超时配置是否合理。

如果使用的是海外模型（如 GPT-4、Claude），网络延迟是主要瓶颈。切换为国内模型（如 Kimi、DeepSeek）通常能将响应时间缩短 50% 以上。如果已使用国内模型但响应仍慢，可检查本机网络连接质量，或参照第12.9.2节适当调大超时阈值以避免因超时导致的重试延迟叠加。

### 12.12.3 群组中无法 @ 机器人

**原因分析**：群组 @ 功能的正常使用需要机器人已加入群组、相关权限已开通，以及（若启用了白名单）群组 ID 已在白名单中。

按以下顺序排查：首先确认机器人已通过第12.7.1节的步骤添加到目标群组，群组中应有机器人加入的系统消息。然后检查 `im:message.group_at_msg` 和 `im:message.group_at_msg:readonly` 两项权限是否已开通并完成版本发布。最后，如果配置了群组白名单，按照第12.8.2节获取该群组的 ID，并将其添加到 `groupAllowlist` 配置中。

### 12.12.4 如何查看 API 消耗？

执行以下命令可查看 API 调用统计信息，用于监控用量和控制成本：

```bash
# 查看 API 调用统计
openclaw stats api

# 在日志中过滤 API 相关记录
tail -f ~/.openclaw/logs/gateway.log | grep "API"
```

统计信息按时间维度（日、周、月）汇总，包括请求次数、token 消耗和估算费用。建议定期查看，及时发现异常消耗。

### 12.12.5 如何备份配置？

配置文件存储了所有凭证和个性化设置，建议在修改配置前及时备份。执行以下命令创建备份：

```bash
# 备份单个配置文件
cp ~/.openclaw/openclaw.json ~/.openclaw/openclaw.json.backup

# 备份整个配置目录（包含日志和缓存）
tar -czf openclaw-backup-$(date +%Y%m%d).tar.gz ~/.openclaw
```

备份文件中包含 App Secret 等敏感信息，请妥善保管，避免上传至公开代码仓库。

## 本章小结

本章介绍了飞书 Bot 的完整配置流程，涵盖从应用创建到生产就绪的全部步骤。

**创建飞书应用**：在飞书开放平台的开发者后台创建企业自建应用，获取 App ID 和 App Secret，并配置必要的消息权限与事件订阅。**配置 OpenClaw**：通过配置向导、手动编辑配置文件或命令行三种方式，将凭证信息写入 OpenClaw 配置，启用飞书频道。**启动并测试**：重启 Gateway 服务使配置生效，通过飞书客户端发送测试消息验证消息收发链路完整性。**访问控制**：根据安全需求配置用户白名单、群组白名单和管理员账号，防止 API 被滥用。**群组配置**：将机器人添加到目标群组，按需选择 @ 提及模式或全部消息模式。**高级功能**：通过多账号配置实现部门隔离，通过多 Agent 配置满足多样化的使用需求。

飞书是 OpenClaw 目前最推荐的 IM 接入渠道。其 WebSocket 长连接机制无需公网地址，配置门槛极低；丰富的消息格式支持使 AI 输出更具可读性；完善的企业账号体系也为权限管理提供了坚实基础。关于其他平台的配置，企微、钉钉和 QQ 的接入方式参见第13章企微钉钉QQ，多平台统一管理的思路参见第11章多平台。
