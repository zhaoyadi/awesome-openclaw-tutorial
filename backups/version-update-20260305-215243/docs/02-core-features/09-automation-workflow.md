# 第9章 自动化工作流

本章目标：学会使用 OpenClaw 构建自动化工作流，实现定时任务、网站监控、日报推送和循环任务配置。自动化工作流是 OpenClaw 区别于普通对话型 AI 助手的核心能力之一——它不需要用户主动触发，而是由系统在指定时间或条件满足时主动发起动作，从而把重复性、周期性的工作交给机器完成，让用户专注于真正需要判断和创造的事情。

- 9.1 定时任务设置
- 9.2 网站监控实战
- 9.3 日报自动推送

## 9.1 定时任务设置

定时任务是自动化工作流最基础的构成单元。本节首先介绍 OpenClaw 实现定时任务的底层机制，然后从简单提醒逐步过渡到周报、月报等复合场景，最后讲解 Cron 表达式，使读者具备自行配置任意时间规则的能力。

### 9.1.1 什么是定时任务

定时任务是指在指定时间自动执行的任务，无需人工干预。与其他对话型 AI 工具不同，OpenClaw 内置了主动发起对话的能力，这使得它能够在用户未打开应用的情况下独立完成工作并推送结果。

OpenClaw 在定时任务方面具备以下核心能力：**心跳机制**可以主动发起对话，无需等待用户输入；**智能调度**能够自动管理多个任务的执行顺序；**灵活配置**支持固定时间、Cron 表达式等多种时间模式；**可靠执行**在任务失败时会自动重试，并在超限后发送错误通知。

### 9.1.2 心跳机制原理

理解心跳机制有助于用户合理规划任务的触发精度。OpenClaw 的守护进程在后台持续运行，每分钟对任务列表执行一次轮询，将当前时间与各任务的触发规则进行匹配，一旦命中则立即执行对应逻辑并将结果推送到指定渠道，同时写入执行日志。由于轮询间隔为1分钟，任务触发存在最多1分钟的延迟，这对日常提醒和报告推送场景完全可以接受。

下图展示了心跳机制的基本结构：

```
┌─────────────┐
│  OpenClaw   │
│   守护进程   │
└──────┬──────┘
       │
       ├─ 每分钟检查一次
       ├─ 匹配定时任务
       ├─ 执行任务逻辑
       └─ 发送消息/执行操作
```

心跳机制的完整工作流程如下：OpenClaw 在系统后台以守护进程形式运行；每分钟触发一次任务列表检查；到达设定时间时自动触发对应任务；执行任务逻辑并将结果发送至目标渠道；最后将本次执行结果写入日志，便于后续查询和排错。

### 9.1.6 Cron 表达式详解

前面的案例均使用自然语言描述执行时间，OpenClaw 会在后台将其转换为 Cron 表达式。理解 Cron 表达式可以帮助用户配置自然语言难以精确表达的复杂时间规则，例如"每隔30分钟"或"工作日每天上午9点"。

Cron 表达式由5个字段组成，各字段之间以空格分隔，从左到右依次表示分钟、小时、日期、月份和星期。每个字段使用数字表示具体值，使用星号（`*`）表示"每个"，使用逗号分隔多个值，使用连字符表示范围，使用斜线表示步长。

```
分钟 小时 日期 月份 星期

* * * * *
│ │ │ │ │
│ │ │ │ └─ 星期 (0-7, 0和7都表示周日)
│ │ │ └─── 月份 (1-12)
│ │ └───── 日期 (1-31)
│ └─────── 小时 (0-23)
└───────── 分钟 (0-59)
```

掌握字段含义后，可以根据以下常用示例快速写出所需的时间规则：

```bash
# 第9章/cron-examples.sh
# 每天早上8点
0 8 * * *

# 每周一上午10点
0 10 * * 1

# 每月1号上午9点
0 9 1 * *

# 每天中午12点和下午6点
0 12,18 * * *

# 每小时执行一次
0 * * * *

# 每30分钟执行一次
*/30 * * * *

# 工作日早上9点
0 9 * * 1-5

# 周末上午10点
0 10 * * 0,6
```

在实际使用中，直接与 OpenClaw 对话即可完成配置，系统会自动推导并展示对应的 Cron 表达式，用户可以据此验证时间规则是否符合预期。以下是一个综合示例：

```
你：帮我设置1个定时任务：
每天早上8点、中午12点、晚上8点提醒我喝水

OpenClaw：好的，开始设置！

Cron 表达式：0 8,12,20 * * *

解释：
- 0：在整点执行
- 8,12,20：早上8点、中午12点、晚上8点
- *：每天、每月、每周

任务内容：
该喝水了！
保持每天8杯水，身体更健康！
今天已喝：[X]杯

已设置定时任务
```

## 9.2 网站监控实战

定时任务解决的是"定时推送"问题，而网站监控解决的是"变化感知"问题。两者在机制上相似，均依赖后台定时检查，但监控任务增加了内容对比和条件过滤步骤：只有当目标页面发生了指定类型的变化，才触发通知。本节将介绍监控原理，并通过博客更新、价格监控、招聘信息三个递进场景演示具体配置方法。

### 9.2.1 为什么需要网站监控

在没有自动化工具的情况下，获取关注网站的最新内容只能依靠手动定期查看，这种方式既耗费时间，又容易因为检查不及时而错过重要信息。网站监控的价值在于把"人主动查找"变成"机器主动推送"，从而做到不遗漏、不延迟。

以下是几种典型的监控需求：技术博客更新时及时获取新文章，用于持续学习；电商商品降价时立即收到通知，把握最佳购买时机；心仪公司发布招聘职位时第一时间投递，提高竞争优先级；行业新闻更新时快速了解动态，保持信息同步；开源项目发布新版本时跟进学习最新特性。

相比手动检查，OpenClaw 的自动监控做到了全天候覆盖、按需推送、内容摘要提取三者的结合，使信息获取效率大幅提升。

### 9.2.2 网站变化检测原理

OpenClaw 的网站监控机制可以概括为四个步骤：

```
定时检查 --> 抓取网页 --> 对比变化 --> 发送通知
```

在具体实现上，OpenClaw 支持三种检测方式。**全文对比**：对整个页面内容进行前后比对，适合结构简单、内容较少的页面；**关键内容对比**：仅针对指定区域（如文章列表、价格标签）进行检测，适合内容复杂的大型页面，可以避免无关内容变化引发误报；**RSS 订阅**：直接解析网站提供的 RSS 源，是最稳定可靠的方式，建议在网站支持 RSS 的情况下优先使用。

了解检测方式有助于用户根据目标网站的特点选择合适的监控策略，从而减少漏报和误报。

### 9.2.3 实战案例1：博客更新监控

博客监控是最常见的监控场景，目标明确——发现新文章就推送标题和摘要。这类场景通常使用全文对比或 RSS 订阅，检查频率设置为每天2次即可满足需求。

**场景描述**：持续监控某个技术博客，每天检查两次，发现新文章后立即推送标题、作者、摘要及预计阅读时长。

### 9.2.4 实战案例2：产品价格监控

与博客监控相比，价格监控增加了条件判断——只有价格低于设定阈值时才发送通知。这要求 OpenClaw 不仅能检测变化，还能对变化内容进行语义理解和数值比较。

**场景描述**：持续监控某商品的价格，每小时检查一次，当价格达到目标值时立即推送购买建议，附带历史价格走势分析。

与 OpenClaw 的对话示例如下：

```
你：帮我监控一款商品的价格，
目标价格：低于5000元就通知我

监控配置：
- 商品：iPhone 15 Pro
- 当前价格：¥5999
- 目标价格：不高于 ¥5000
- 检查频率：每小时一次
- 通知方式：飞书消息 + 短信

价格历史：
- 最高价：¥6999（2026-01-01）
- 最低价：¥5499（2026-01-20）
- 平均价：¥5899

预测：
根据历史数据，预计2周内可能降至目标价格

已设置监控任务
```

当价格达到目标条件时，推送内容如下：

```
OpenClaw：价格提醒！

商品：iPhone 15 Pro
当前价格：¥4999

价格变化：
- 原价：¥5999
- 现价：¥4999
- 降价：¥1000（16.7%）

近期最低价
建议立即购买

是否需要协助跳转购买页面？
```

价格监控的关键在于设置合理的检查频率。电商价格可能每天多次变化，每小时检查一次既能保证及时性，又不至于给目标网站造成过多请求压力。

### 9.2.6 监控频率优化

三个监控案例使用了不同的检查频率，这并非随意设定，而是根据目标内容的更新规律和对时效性的要求综合权衡的结果。不同类型的监控任务推荐频率见表9-2。

表9-2  不同监控类型的推荐频率

| 监控类型     | 推荐频率     | 理由             |
| ------------ | ------------ | ---------------- |
| 博客更新     | 每天2次      | 更新不频繁       |

## 9.3 日报自动推送

网站监控解决的是"特定内容变化时通知我"的问题，而日报推送解决的是"每天定时向我汇总多源信息"的问题。两者都属于信息聚合，区别在于日报更加主动、结构更加固定，且往往需要从多个数据源同时采集内容。本节将介绍日报的设计思路，并通过 AI 行业日报、技术资讯日报、个人工作日报三个场景演示完整的配置和推送效果。

### 9.3.1 为什么需要日报

一份设计合理的日报能够在每天固定时间将分散在各处的信息汇聚成一份结构化摘要，帮助用户快速完成信息扫描，而无需逐一打开各类应用。日报的核心价值体现在以下几个方面。

**信息聚合**：将行业新闻、GitHub 动态、技术文章、日历安排等来自不同来源的信息合并到同一份报告中，一次性完成浏览，减少应用切换的时间损耗。**保持更新**：通过固定的推送节奏，帮助用户形成定期了解行业动态的习惯，避免因遗忘检查而错过重要信息。**知识积累**：日报中的精选文章和技术动态以结构化形式呈现，便于保存到知识库（详见第7章），逐步形成可检索的个人知识体系。**工作规划**：将当天日程和待办事项纳入日报，使用户在正式开始工作前就对全天安排有清晰认知。

### 9.3.2 日报内容设计

一份完整日报的内容设计应当兼顾外部信息和内部安排两个维度，前者包括行业动态、技术文章、开源项目，后者包括日程提醒和待办事项。以下是一个通用的日报结构模板，可以根据个人需求增减模块：

```markdown
# 行业日报（2026-02-11）

## 今日要闻（3-5条）
- 重要新闻
- 行业动态
- 技术突破

## 热门话题（3-5个）
- 社区讨论
- 技术趋势
- 产品发布

## 精选文章（3-5篇）
- 技术教程
- 深度分析
- 实战案例

## 开源项目（2-3个）
- 新项目推荐
- 项目更新
- 实用工具

## 今日日程
- 会议安排
- 任务提醒
- 重要事项

## 每日一句
- 技术感悟
```

模块数量和每个模块的条目数可以根据实际阅读习惯调整。内容过多会导致日报冗长难以快速消化，建议每个模块保持3到5条，总阅读时间控制在5分钟以内。

## 9.4 循环任务配置

定时任务和日报推送都属于"在固定时间点触发一次"的模式，而循环任务则是"以固定间隔或特定条件持续运行"的模式。两者的核心区别在于：定时任务关注的是"何时执行"，循环任务关注的是"如何持续执行"以及"任务之间如何协作"。本节将介绍循环任务的类型，并通过数据备份、健康提醒、任务依赖管理三个场景展示其应用方式。

### 9.4.1 什么是循环任务

循环任务是指按照一定规律重复执行的任务。与定时任务在固定时间点触发不同，循环任务以固定的时间间隔连续执行，适用于需要持续监控或周期性操作的场景。

定时任务与循环任务的区别见表9-4。

表9-4  定时任务与循环任务对比

| 特性     | 定时任务           | 循环任务           |
| -------- | ------------------ | ------------------ |
| 执行时间 | 固定时间点         | 固定间隔           |
| 示例     | 每天上午8点        | 每隔2小时          |
| 适用场景 | 日报、固定提醒     | 监控、周期性检查   |

### 9.4.2 循环任务类型

**固定间隔型**：以指定的时间间隔重复执行，例如每隔30分钟、每隔6小时。时间间隔在任务创建时确定，运行期间保持不变，适用于数据备份、状态检查等需要规律性执行的场景。

**条件触发型**：在满足特定条件时才执行后续逻辑，例如"当天气预报显示降雨时提醒带伞"或"当任务队列为空时发送完成通知"。这类任务通常与固定间隔结合使用：系统按固定频率检查条件是否满足，满足时执行对应操作。

**依赖任务型**：多个任务按顺序排列，前一个任务完成后自动触发下一个任务，形成工作流链路。这种模式适合流程性工作，可以确保各步骤按正确顺序执行，并在任意步骤失败时中断整个流程，避免错误传播。

### 9.4.3 实战案例1：数据备份

数据备份是固定间隔型循环任务最典型的应用场景。重要数据需要定期保存副本，间隔时间应当短于可接受的最大数据丢失窗口。

**场景描述**：每隔6小时自动备份一次备忘录和日历数据，每天执行4次（00:00、06:00、12:00、18:00），保留最近7天的备份记录，超出时自动清理旧版本。

### 9.4.5 任务依赖管理

任务依赖管理将多个独立任务串联成完整的工作流，是循环任务配置中最复杂也最实用的部分。当每个步骤都有明确的前置条件时，使用依赖链而非单独配置每个任务，可以确保逻辑的完整性和一致性。

**场景描述**：每天早上7点执行一个由三个步骤组成的任务链：首先检查天气，然后根据天气结果决定是否发送特殊提醒（下雨提醒带伞、气温低于10度提醒加衣），最后推送当天日程，三个步骤按顺序执行，前一步骤完成后自动触发下一步骤。

与 OpenClaw 的对话示例如下：

```
你：帮我设置任务链：

1. 每天早上7点，检查天气
2. 如果下雨，提醒带伞
3. 如果气温低于10度，提醒多穿衣服
4. 然后推送今日日程

OpenClaw：好的，开始设置任务链！

任务链配置：

任务1：检查天气
- 执行时间：每天 07:00
- 数据来源：天气API
- 下一步：根据天气决定

任务2：天气提醒（条件触发）
- 触发条件：下雨 或 气温<10°C
- 提醒内容：
  - 下雨 --> 记得带伞
  - 气温低 --> 多穿衣服

任务3：推送日程
- 执行时机：任务2完成后
- 推送内容：今日日程安排

已设置任务链
首次执行：明天 07:00
```

任务链的价值在于将条件判断封装在工作流内部，用户无需手动处理每种天气情况，系统会自动根据实时数据选择合适的提醒内容，最终统一推送日程。这种模式可以扩展到更复杂的业务场景，例如第11章中介绍的多平台消息协同分发。

### 9.4.6 错误重试机制

循环任务在长期运行过程中不可避免地会遇到网络中断、数据源不可用等故障情况。OpenClaw 内置了梯度重试机制，使单次故障不会导致任务永久失败。重试策略采用指数退避方式，每次重试的等待时间逐步增加，在减少无效请求的同时给故障系统留出恢复时间。

重试流程如下：

```
任务执行失败 --> 等待1分钟 --> 重试
重试失败     --> 等待5分钟 --> 再次重试
再次失败     --> 等待15分钟 --> 最后重试
最后失败     --> 发送错误通知 --> 停止任务
```

当三次重试均失败后，系统将发送错误通知，示例如下：

```
OpenClaw：任务执行失败

任务名称：AI行业日报
失败时间：2026-02-11 08:00
失败原因：网络连接超时

已尝试重试：3次
重试结果：全部失败

建议：
1. 检查网络连接
2. 检查数据源是否可用
3. 手动执行任务

是否需要继续重试？
```

错误通知提供了失败原因和排查建议，用户可以根据提示快速定位问题。对于间歇性网络故障，选择"继续重试"通常可以在网络恢复后自动完成任务；对于数据源本身的故障，则需要先检查数据源状态再决定是否手动触发。

## 常见问题

**Q1：定时任务没有在预期时间执行？**

检查 OpenClaw 守护进程是否正常运行，可以在设置中查看进程状态。其次查看任务日志，确认任务是否被记录为已触发但执行失败，还是完全未被触发。如果完全未触发，通常是守护进程未启动或任务配置中的时间规则存在语法错误。

**Q2：网站监控任务被目标网站封禁访问？**

降低监控频率至每天2至3次，并在任务配置中添加随机延迟（例如在整点前后随机偏移0至10分钟）。如果目标网站提供 RSS 源，优先改用 RSS 方式，这样既稳定又不会给目标服务器造成额外负担。

**Q3：日报中的内容与实际不符或数据来源信息有误？**

检查各数据源的访问配置是否有效，确认相关账号授权是否已过期。如果是日历或备忘录数据来源，需要确认 OpenClaw 是否具有对应系统权限。对于外部网站数据，可以尝试手动触发一次任务并查看原始抓取结果，以判断问题出在抓取阶段还是内容处理阶段。

**Q4：循环任务长期运行后系统资源占用增高？**

检查是否有任务陷入循环重试而未正常终止。优化任务逻辑，减少每次执行时的不必要操作，例如避免在每次检查时重新加载大量历史数据。对于低频任务，适当增大执行间隔；对于高频任务，评估是否真正需要如此频繁的执行。

**Q5：如何暂停或删除某个任务？**

直接告知 OpenClaw 暂停或删除指定任务名称即可。暂停操作会保留任务配置，后续可以随时恢复；删除操作会彻底移除任务，需要重新配置才能恢复。建议在修改任务参数时选择暂停而非删除，以保留原始配置作为参考。
